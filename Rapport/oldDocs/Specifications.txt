  1. PROJECT OVERVIEW
================================================================================

Project Name: SecureBank AI
Description: A secure, web-based banking and wallet management platform with
             AI-powered threat detection, attack prevention, and real-time
             security monitoring for system administrators.

Core Vision:
  - Provide secure, user-friendly banking services
  - Deploy advanced AI-based security threat detection
  - Enable administrators to monitor, analyze, and respond to attacks
  - Offer real-time visibility into security events via Grafana dashboards
  - Maintain 100% open-source, free, and locally-hosted AI models

Target Users:
  - End Users: Customers managing accounts and transfers
  - Administrators: Security teams monitoring and responding to threats
  - DevOps: Infrastructure teams managing deployment and monitoring

  4. AI/ML SECURITY COMPONENTS
================================================================================

4.1 THREAT DETECTION ENGINE
---------------------------
  Purpose: Real-time identification of suspicious activities

  Data Inputs Monitored:
    - Source IP Address
    - Login Attempt Frequency (attempts/minute)
    - Geographic Location (country, city)
    - Access Time (hour of day, day of week)
    - Device Information (browser, OS, user agent)
    - Failed Login Patterns
    - Query Patterns (detecting SQL injection)
    - Transaction Velocity (unusual amounts or frequency)
    - Account Access Anomalies

  AI Model: Ollama + Mistral 7B or LLaMA 3
    - Analyzes behavioral patterns in real-time
    - Compares against baseline user profiles
    - Generates risk scores (0-100)
    - Produces actionable threat descriptions

  Anomaly Score Calculation:
    Normal User Profile (Baseline):
    {
      "avg_login_time_UTC": 9,
      "login_frequency": "5/day",
      "typical_locations": ["US", "CA"],
      "typical_devices": ["Chrome on Windows", "Safari on iOS"],
      "avg_transaction_amount": 500,
      "transaction_frequency": "10/week"
    }

    Real-Time Event:
    {
      "ip": "185.220.101.xx",
      "attempts": "20/min",
      "location": "Unknown (Proxy/VPN)",
      "time": "3:00 AM",
      "device": "Bot User Agent",
      "login_failures": 15,
      "transaction_attempts": "50/min high amounts"
    }

    Risk Assessment: 95/100 (CRITICAL)
    Recommendation: BLOCK IP, ALERT ADMIN, LOCK ACCOUNT

4.2 BEHAVIORAL ANOMALY DETECTION
---------------------------------
  Model: Isolation Forest + Autoencoder (HuggingFace transformers)
  Purpose: Identify deviations from user baseline behavior

  Detected Anomalies:
    - Impossible travel (login from 2 different continents < 2 hours)
    - Time-based anomalies (access at unusual hours)
    - Device fingerprint changes (new device from known user)
    - Amount-based anomalies (transfer amount 10x higher than average)
    - Velocity anomalies (too many transactions in short period)
    - Query pattern anomalies (SQL injection attempts, malformed requests)

4.3 IP REPUTATION & GEOLOCATION
-------------------------------
  Data Sources:
    - MaxMind GeoIP2 Lite (free tier)
    - AbuseIPDB API (free tier, 15 requests/day)
    - Custom IP blacklist (admin-managed)
    - VPN/Proxy detection (IP2Proxy free database)

  Risk Factors:
    - Unknown VPN/Proxy: +20 risk points
    - High-risk country: +15 risk points
    - Previous attack source: +50 risk points
    - Datacenter IP: +10 risk points

4.4 SQL INJECTION DETECTION
---------------------------
  Approach: Rule-based + ML-based hybrid detection
  
  Rule-Based Detection:
    - Pattern matching for common SQL injection payloads
    - Special character sequences: '; DROP --, UNION SELECT
    - Encoding detection: URL, hex, unicode bypass attempts
    - Comment patterns: /*, --, #, ;

  ML-Based Detection (Ollama + CodeLlama):
    - Parse SQL queries against normalized AST
    - Compare against known malicious query patterns
    - Flag suspicious parameter types and counts
    - Suggest query alternatives to developers

================================================================================
  6. SECURITY ARCHITECTURE
================================================================================

6.1 AUTHENTICATION & AUTHORIZATION
----------------------------------
  User Authentication:
    - Google OAuth 2.0 via Firebase Auth (free tier)
    - JWT tokens for API requests (15 min expiry)
    - Refresh tokens (7 day expiry, stored in httpOnly cookie)
    - Multi-Factor Authentication optional (TOTP)

  Role-Based Access Control (RBAC):
    Role          | Permissions
    --------------|------------------------------------------------------------------
    User          | View own account, transactions, profile; initiate transfers
    Admin         | View all dashboards, manage IPs, generate reports, user lockdown
    SuperAdmin    | Full system access, user/admin management, config changes
    Audit         | Read-only access to logs (no write/delete permissions)

6.2 DATA PROTECTION
-------------------
  Encryption:
    - TLS 1.3 for all in-transit data
    - AES-256 encryption for sensitive fields at rest
      * Passwords: bcrypt with 12 salt rounds
      * API keys: encrypted with master key
      * PII: encrypted field-level using libsodium
    - Field-level encryption for transaction details

  API Security:
    - CORS policy: strict domain whitelist
    - CSRF protection: token-based validation
    - Rate limiting: 100 requests/min per IP
    - Input validation: strict type checking, SQL parameterization
    - Output encoding: HTML entity encoding for JSON responses

  Logging & Audit:
    - All admin actions logged (timestamp, action, user)
    - All failed login attempts logged (IP, user account, time)
    - All data access logged (Read-only for audit role)
    - Logs stored in tamper-proof format (immutable append-only)
    - Log retention: 90 days online, 1 year archived

  9. DATA MODELS
================================================================================

9.1 USER/ACCOUNT SCHEMA
-----------------------
  User Table:
    - id (UUID primary key)
    - email (unique, indexed)
    - name
    - google_id
    - profile_picture_url
    - created_at
    - updated_at
    - last_login_at
    - total_login_count
    - mfa_enabled (boolean)
    - mfa_secret (encrypted)
    - account_status (active, locked, suspended)

  Account Table (Wallet):
    - id (UUID)
    - user_id (foreign key)
    - account_number
    - account_type (checking, savings)
    - currency (USD, EUR, etc.)
    - balance (decimal)
    - created_at
    - updated_at

  Transaction Table:
    - id (UUID)
    - from_account_id (foreign key)
    - to_account_id (foreign key)
    - amount (decimal)
    - currency
    - timestamp
    - status (pending, completed, failed, cancelled)
    - transaction_hash (for immutability)

9.2 SECURITY/AUDIT SCHEMA
--------------------------
  LoginEvent Table:
    - id (UUID)
    - user_id (foreign key)
    - ip_address
    - user_agent
    - geolocation_country
    - device_fingerprint_hash
    - success (boolean)
    - failure_reason (if failed)
    - timestamp
    - is_flagged_by_ai (boolean)
    - is_blocked (boolean)

  AttackEvent Table:
    - id (UUID)
    - attack_type (enum: brute_force, sqli, impossible_travel, etc.)
    - source_ip
    - target_user_id (if known)
    - payload_sample (encrypted)
    - risk_score (0-100)
    - ai_analysis (JSON, LLM output)
    - action_taken (blocked, alerted, logged)
    - timestamp
    - admin_notes (text)
    - status (open, investigating, resolved)

  BlockList Table:
    - id (UUID)
    - ip_address (indexed)
    - reason
    - added_by (admin user_id)
    - added_at
    - expires_at (NULL for permanent)
    - is_auto_generated (boolean)

  AuditLog Table:
    - id (UUID)
    - admin_id
    - action (enum: login, logout, add_blocklist, remove_blocklist, etc.)
    - resource_type (User, IP, Transaction, etc.)
    - resource_id
    - changes (JSON diff)
    - timestamp
    - ip_address

  11. SECURITY COMPLIANCE & BEST PRACTICES
================================================================================

11.1 DATA PRIVACY
-----------------
  - No PII stored in logs except hashed identifiers
  - User consent for data processing (GDPR if EU)

----------------------
