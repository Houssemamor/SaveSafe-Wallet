================================================================================
  PROJECT SUMMARY
================================================================================

Name: SaveSafe Wallet (SSW)
Type: Secure web-based banking & wallet platform with AI anti-pirating system

Core Vision:
  - Secure, user-friendly banking and wallet management
  - AI-powered threat detection using only free/open-source models
  - Real-time admin security dashboard via Grafana
  - Microservices architecture with async messaging (Kafka)
  - 100% open-source stack, no paid APIs

Target Users:
  - End Users: manage accounts, wallets, transfers
  - Administrators: monitor security, respond to threats
  - System IA: autonomous threat detection and scoring

================================================================================
  TECH STACK
================================================================================

Frontend     : Angular (TypeScript)
Backend      : .NET 8 (C#, microservices)
Database     : PostgreSQL (users, transactions, logs)
Cache        : Redis (sessions, rate-limiting, risk scores)
Messaging    : Apache Kafka (inter-service events, Outbox pattern)
AI/ML        : Ollama (Mistral 7B, LLaMA 3) for risk scoring
               Isolation Forest + Autoencoders (scikit-learn, HuggingFace)
               CodeLlama for SQL injection detection
               sentence-transformers (embeddings)
Auth         : JWT (access + refresh tokens), Google OAuth 2.0
Monitoring   : Grafana + Prometheus (dashboards, metrics)
Logging      : Loki (centralized logs)
Containers   : Docker + Docker Compose (local), Kubernetes (prod)
Cloud        : AWS free tier
CI/CD        : GitHub Actions
Collaboration: GitHub (PRs, branches), Jira (sprints)

================================================================================
  MICROSERVICES
================================================================================

  1. API Gateway       - routing, rate limiting, CORS, request validation
  2. Auth Service      - register, login, JWT, refresh, RBAC, Google OAuth
  3. User Service      - profile CRUD, preferences, account deletion
  4. Wallet Service    - wallet creation, balance, ledger (credit/debit), history
  5. Payment Service   - Stripe checkout, webhooks, reconciliation, idempotency
  6. Security Service  - event capture, IP blocklist, SQLi detection, audit trail
  7. AI Risk Service   - risk scoring, anomaly detection, policy actions (async)
  8. Notification Svc  - email templates, retry/backoff, delivery journal
  9. Reporting Service - payment aggregates, wallet KPIs, admin dashboard data

Communication:
  - Sync: REST (JSON) between Gateway and services
  - Async: Kafka topics for events (payment.*, security.event.*, risk.score.*)
  - Outbox pattern: outbox_events table -> Kafka publisher worker
  - Dead-letter topics (DLT) for failed consumers

  [ ] POST /api/auth/refresh
      - Validate refresh token
      - Issue new JWT
  [ ] POST /api/auth/logout
      - Invalidate refresh token
  [ ] RBAC middleware
      - Roles: User, Admin
      - Attribute-based authorization on controllers
      - Reject unauthorized requests with 403

STEP 4: Wallet Service (basic)
  [ ] Auto-create wallet when user registers (triggered by Auth Service)
  [ ] GET /api/wallet/balance
      - Return current balance for authenticated user
  [ ] Ledger: append-only table recording every credit/debit

################################################################################
  SPRINT 3 (2 weeks) - PAYMENTS & NOTIFICATIONS
  Goal: Stripe checkout + webhooks + notifications + full ledger
################################################################################

STEP 1: Payment Service (Stripe Integration)
  [ ] POST /api/payment/checkout-session
      - Input: serviceType, amount
      - Create Stripe CheckoutSession
      - Save transaction record with status PENDING
      - Return Stripe session URL to frontend
  [ ] POST /api/payment/webhook
      - Verify Stripe signature (webhook secret)
      - On checkout.session.completed -> update transaction to SUCCESS
      - On checkout.session.expired -> update transaction to FAILED
      - Credit wallet via ledger entry on success
      - Publish Kafka event: payment.completed or payment.failed

STEP 2: Kafka Event Publishing (Outbox Pattern)
  [ ] Create outbox_events table (id, topic, payload, status, created_at)
  [ ] Payment Service writes to outbox_events instead of directly to Kafka
  [ ] Background worker: poll outbox_events WHERE status=PENDING
      - Publish to Kafka topic
      - Mark as SENT on success, FAILED on error
      - Retry with backoff for FAILED events

STEP 3: Notification Service
  [ ] Kafka consumer: listen to payment.completed, payment.failed topics
  [ ] Email template: payment success (amount, date, transaction ID)
  [ ] Email template: payment failed (reason, retry suggestion)
  [ ] Notification record: id, type, recipient, status, retry_count, sent_at
  [ ] Retry with exponential backoff (max 3 retries)
  [ ] Journal all notifications (sent, failed, retried)

STEP 4: Idempotency & Reconciliation
  [ ] Idempotency key on checkout-session to prevent double payments
  [ ] Idempotent consumer: check event_id before processing Kafka message
  [ ] Daily reconciliation job: compare Stripe dashboard vs local transactions
      - Flag mismatches for admin review

STEP 5: Angular Frontend - Payment Flow
  [ ] Payment page: select service, enter amount, click "Pay"
  [ ] Redirect to Stripe Checkout
  [ ] Success/cancel return pages
  [ ] Updated dashboard: show payment transactions in wallet history
  [ ] Notification preferences in profile settings

STEP 6: Testing
  [ ] Unit tests: checkout session creation, webhook signature verification
  [ ] Integration test: full flow (create session -> webhook -> ledger update -> notification)
  [ ] Test idempotency: submit same payment twice, verify single charge

DELIVERABLES:

  - Outbox pattern with Kafka publishing
  - Notification service consuming payment events, sending emails

################################################################################
  SPRINT 4 (2 weeks) - SECURITY & AI
  Goal: Security service + AI risk scoring + monitoring
################################################################################

STEP 1: Security Service - Event Collection
  [ ] Capture on every login: ip, user_agent, country (GeoIP), timestamp, success/fail
  [ ] Store in security_events table
  [ ] Add correlation_id to all requests (middleware, propagated via headers)
  [ ] Rate-limit via Redis:
      - 5 failed logins/min per IP -> temporary block
      - 100 requests/min per IP -> rate limit response (429)

STEP 2: Security Service - IP Blocklist & SQLi Detection
  [ ] BlockList table: ip, reason, added_by, expires_at, is_auto
  [ ] GET /api/admin/blocklist - list blocked IPs
  [ ] POST /api/admin/blocklist - manually block IP
  [ ] DELETE /api/admin/blocklist/:ip - unblock
  [ ] Middleware: check incoming IP against blocklist (Redis cache for speed)
  [ ] SQL Injection detection (rule-based):
      - Pattern match: '; DROP, UNION SELECT, OR 1=1, etc.
      - Log detected attempts as AttackEvent
      - Block IP if repeated attempts

STEP 3: Security Service - Immutable Audit Trail
  [ ] audit_log table: id, admin_id, action, resource_type, resource_id,
      prev_hash, curr_hash, timestamp
  [ ] Each row: curr_hash = SHA256(prev_hash + action + timestamp + data)
  [ ] Hash chain makes tampering detectable
  [ ] GET /api/admin/security/timeline - paginated security event timeline

STEP 4: AI Risk Service (Async via Kafka)
  [ ] Kafka consumer: listen to security.event.created
  [ ] Risk scoring logic:
      - Rule-based baseline:
        * Failed login: +20 pts
        * Unknown VPN/Proxy: +15 pts
        * Unusual hour (midnight-5am): +10 pts
        * New device: +15 pts
        * High-risk country: +15 pts
        * Impossible travel: +40 pts
      - ML model (Isolation Forest): train on historical login events
        * Features: login_hour, country_code, device_hash, attempts_count
        * Output: anomaly score
      - Ollama (Mistral 7B): analyze event context, generate threat description
  [ ] Publish Kafka event: risk.score.computed (score, action recommendation)
  [ ] Action policies:
      - Score 0-20: ALLOW
      - Score 21-50: CHALLENGE (require 2FA or email verification)
      - Score 51-80: TEMP_BLOCK (block for 1 hour, alert admin)
      - Score >80: BLOCK (permanent until admin review)
  [ ] Fallback: if AI service is down, use rule-based scoring only

STEP 5: Security Service - React to AI Scores
  [ ] Consume risk.score.computed events
  [ ] Apply action: update blocklist, lock account, send alert to admin
  [ ] Alert admin via Notification Service on critical scores (>80)

STEP 6: Admin Dashboard (Angular)
  [ ] Security overview page:
      - Attacks today (count, trend)
      - Recent attack timeline (table with IP, type, risk score, action)
      - Blocked IPs list with unblock action
  [ ] User management page:
      - List users, search/filter
      - Lock/unlock account
      - View user's login events
  [ ] IP management page:
      - Current blocklist
      - Add/remove manually
      - View IP reputation (risk score, country, VPN status)

STEP 7: Grafana & Prometheus Setup
  [ ] Prometheus: scrape metrics from each .NET service (/metrics endpoint)
      - request_count, request_duration, error_count per service
      - login_success_total, login_failure_total
      - kafka_messages_published, kafka_messages_consumed
  [ ] Grafana dashboards:
      - Security: attacks/day, blocked IPs, failed logins, SQLi attempts
      - Payments: transactions/day, success rate, volume
      - System: CPU, RAM, request latency, error rate
  [ ] Loki: centralized log aggregation from all services
  [ ] Alert rules: notify admin email on critical thresholds

STEP 8: Testing
  [ ] Unit tests: risk scoring logic, blocklist operations, hash chain integrity
  [ ] Integration test: login -> security event -> Kafka -> AI scoring -> block
  [ ] Test Grafana dashboards with sample data

DELIVERABLES:

================================================================================
  SECURITY ARCHITECTURE NOTES
================================================================================

Authentication:
  - JWT: 15 min access token, 7 day refresh token (httpOnly cookie)
  - Google OAuth 2.0 via Firebase Auth or direct Google API
  - RBAC: User, Admin roles enforced at middleware level
  - Optional TOTP-based 2FA

Data Protection:
  - TLS 1.3 for all traffic
  - bcrypt (12 rounds) for passwords
  - AES-256 for sensitive fields at rest
  - Field-level encryption for PII
  - SQL parameterization everywhere (no raw queries)

Rate Limiting:
  - 100 req/min per IP (general)
  - 5 failed logins/min per IP (auth-specific)
  - Redis-backed counters

Logging:
  - All admin actions logged (append-only, hash-chained)
  - All login events logged (IP, user_agent, country, success/fail)
  - No PII in general logs (masked)
  - 90 days online retention, 1 year archived
